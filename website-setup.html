Building a Personal Website on GitHub Pages

A step-by-step record of how erikgartland.com was set up — from an empty repository to a mobile-friendly, search-indexed site on a custom domain.

1

Set up the GitHub repository

GitHub Pages serves a website directly from a repository, for free. The repository name determines your default web address.

  1. Create a new repository named username.github.io (for this site, erik-gartland.github.io). The name must match your GitHub username exactly.
  2. Make the repository public — GitHub Pages won't serve private repos on the free tier.
  3. Add an index.html file at the root. This becomes your homepage.

Why this name

When a repo is named username.github.io, GitHub treats it as your primary user site and serves it at that address automatically, rather than as a sub-path like username.github.io/project.

2

Enable GitHub Pages

Once the repo has content, point GitHub Pages at the branch you want to publish.

  1. Go to the repository's Settings → Pages.
  2. Under Source, choose Deploy from a branch.
  3. Select the main branch and the / (root) folder, then save.
  4. Wait a minute or two — GitHub builds the site and shows a live link at the top of the Pages settings.

From here on, every time you push a change to main, the live site rebuilds automatically.

3

Connect a custom domain

A custom domain (erikgartland.com) replaces the default github.io address. This involves two sides: telling your domain registrar where to point, and telling GitHub which domain to expect.

On the registrar (Namecheap)

Add DNS records pointing your domain to GitHub's servers. For an apex domain (no www), add four A records pointing to GitHub's IP addresses:

185.199.108.153
185.199.109.153
185.199.110.153
185.199.111.153

Then add a CNAME record for the www subdomain pointing to username.github.io.

On GitHub

  1. In Settings → Pages → Custom domain, enter erikgartland.com and save. This creates a CNAME file in your repo.
  2. Once DNS propagates (can take up to a few hours), check Enforce HTTPS so the site loads securely.

Heads up

DNS changes are not instant. If the domain doesn't connect right away, give it a few hours before troubleshooting — propagation delay is the usual culprit, not a misconfiguration.

4

Remove .html from URLs

Clean URLs like /research/ read better and are easier to share than /research.html. The trick is to use folders with index files.

Instead of a flat file like research.html, create a folder named research/ containing a single index.html. A web server automatically serves index.html when someone visits the folder, so /research/ just works.

research/index.html   →  served at  /research/
about/index.html      →  served at  /about/
cv/index.html         →  served at  /cv/
media/index.html      →  served at  /media/

Audit your internal links

After restructuring, update every internal link to use the clean path (href="/research/" rather than href="research.html"). A single missed link can break navigation or send visitors to a dead page, so check the nav bar on every page.

Watch out

If old .html URLs were already indexed by search engines, they'll now 404. Step 07 covers how to redirect them and clear them from search results.

5

Build a design system

A small, consistent set of colors and fonts — defined once and reused everywhere — keeps every page feeling like part of the same site.

Define tokens as CSS variables

Declaring colors and fonts as variables in :root means a single change updates the whole page. This site's palette:

  • Seafoam green — primary accent
  • Steel blue — secondary
  • Soft coral — highlight accent
  • Cream & charcoal — neutral background and text

And three typefaces, each with a job: Lora for display headings, Figtree for body text, and Fira Code for labels and small-caps detail.

:root {
  --seafoam-500: #2d9e72;
  --charcoal: #2a2f2e;
  --cream: #f7f3ed;
  --f-display: 'Lora', serif;
  --f-body: 'Figtree', sans-serif;
  --f-mono: 'Fira Code', monospace;
}

Structural language

Beyond color and type, consistency comes from repeated structural devices: a mono-font eyebrow above each section title, hairline dividers between entries, and small uppercase labels. These cues are reused on every page so the site reads as one coherent whole.

6

Make it mobile-friendly

Most visitors arrive on phones. A handful of CSS media queries adapt the desktop layout to narrow screens without maintaining a separate mobile site.

The viewport meta tag

Every page needs this in the <head> so mobile browsers render at the correct scale:

<meta name="viewport" content="width=device-width, initial-scale=1.0">

Media queries for the breakpoint

Wrap mobile-specific rules in a media query. Below 680px wide, this site:

  • Collapses multi-column grids (research areas, skills) to a single column
  • Reduces section padding so content isn't lost to whitespace
  • Stacks the hero headshot above the text, centered
  • Stretches buttons to full width for easier tapping

A hamburger menu for navigation

When the nav links don't fit across a phone screen, they're hidden and replaced by a hamburger button that toggles a dropdown. The button is hidden on desktop and only appears below the breakpoint:

@media (max-width: 680px) {
  .nav-hamburger { display: flex; }
  .nav-links { display: none; }
}

Test as you go

Resize your browser window narrow, or use your browser's device-preview mode, to catch layout issues before publishing. What looks fine at desktop width can crowd or overflow on a phone.

7

Set up SEO

Search engine optimization helps people find the site and ensures search results point to the right, current URLs.

Per-page metadata

Each page should have a unique <title>, a meta description, a canonical link, and Open Graph tags so links preview nicely when shared. Structured data (JSON-LD) can further describe who you are to search engines.

Sitemap and robots

A sitemap.xml at the root lists every canonical URL so search engines know exactly what to index. A robots.txt tells crawlers they're welcome and points them to the sitemap. Submit the sitemap in Google Search Console to speed up discovery.

Handling old URLs after a restructure

When clean URLs (Step 04) made the old .html paths return 404s, two things cleared them from search results:

  1. In Google Search Console: use the Removals tool to suppress each old URL, then URL Inspection → Request Indexing on each correct new URL.
  2. A redirecting 404 page: a custom 404.html at the root with a small script that catches known old paths and forwards them to the new location — a safety net for anyone with an old bookmark.
const redirects = {
  '/research.html': '/research/',
  '/about.html':    '/about/',
  '/cv.html':       '/cv/'
};
const path = window.location.pathname;
if (redirects[path]) {
  window.location.replace(redirects[path]);
}

Be patient

Search engines recrawl on their own schedule. Even after everything is correct, it can take days to weeks for old results to drop and new ones to settle in.

8

Add a favicon

The favicon is the little icon in browser tabs and next to search results. Google is particular about how it's served.

A favicon can be embedded directly in HTML as a base64 data URI, but Google won't reliably display base64 favicons in search results. The fix is to serve it as a standalone file:

  1. Save the icon as favicon.png at the root of the repo.
  2. Reference it in every page's <head>:
<link rel="icon" type="image/png" href="/favicon.png">

Verify it's reachable by visiting yourdomain.com/favicon.png directly. Once Google can fetch it there, it'll appear in results on the next crawl.

On base64 data URIs

A data URI embeds a file's binary content as a long text string right in the HTML, avoiding a separate network request. Handy for tiny assets, but search engines treat a real, fetchable file more reliably — which is why the favicon lives as its own file.

9

Make ongoing edits

Day-to-day changes don't require any local tooling — GitHub's web editor is enough.

  1. Navigate to the file you want to change in the GitHub repo.
  2. Click the pencil icon to edit in the browser.
  3. Paste in the updated content and commit. The live site rebuilds within a minute or two.

A few hard-won lessons

  • Hover and color effects can silently fail when a CSS rule is missing a matching :hover or transition on the right class. The fix is usually at the class level, not the element.
  • Browser caching can mask changes. If an update doesn't appear, hard-refresh (Cmd/Ctrl + Shift + R) or append ?v=2 to the URL to bypass the cache.
  • Restructuring URLs means auditing every link. One overlooked href can quietly break navigation.

Keep documents in sync

If the site links to a downloadable resume or CV PDF, remember those are separate files. Updating a link on the site doesn't change the PDF — re-export and re-upload it separately when its contents change.
